 


1. 「未登录用户点击需要账号资产的操作时，调起登录 / 注册弹窗」要改

你现在文档第八部分写了：

未登录用户点击需要账号资产的操作时，调起登录 / 注册弹窗。

这和我们定的“游客可以在 App 内进行任何操作，并可以产生资产”冲突。

应该改成：

游客状态下，用户可以正常进行 Portfolio、Wishlist、Scan、Search 快捷收藏、Collection Item 编辑等资产相关操作，不强制登录。
游客资产通过 anonymous_account 绑定并服务端保存。用户注册新账号时，游客资产迁移到新账号；用户登录已有账号时，游客资产不自动迁移，App 切换展示已有账号资产。

也就是说：资产操作不拦登录，只有注册/登录入口本身才进入账号流程。

2. 游客资产这块现在方向对，但要把“本地游客资产”措辞统一改掉

你文件里第十四部分前面还有一些类似：

当前设备上的游客资产
本地游客资产
保留游客本地资产

但后面第十五部分又写了游客资产通过后端 anonymous_account 同步、存储于后端 D1 数据库，旧的“仅本地、不跨设备、卸载丢失”已作废。 

建议统一成：

游客资产绑定 anonymous_account，并在服务端备份。客户端本地可缓存游客资产，但服务端 anonymous_account 为游客资产的主数据来源。

然后把这些旧说法替换：

现在写法	建议改成
本地游客资产	游客资产 / anonymous_account 资产
当前设备上的游客资产	当前 anonymous_account 下的游客资产
不删除本地游客数据	不删除 anonymous_account 资产及本地缓存
游客资产保持不变	游客资产仍保留在原 anonymous_account 下

3. 登录已有账号后，游客资产“无变动”还要补一句“切回游客时可恢复”

你现在写了：

游客状态下登录已有账号时，不自动迁移当前游客资产。登录成功后 App 切换为该账号资产数据；游客资产保持不变，不与已有账号资产合并。 

这个是对的，但还差一句关键规则：

登录已有账号后，原 anonymous_account 资产仍保留。用户退出登录后，如果客户端仍持有该 anonymous_account 绑定关系，则切回游客状态并展示该游客资产。

否则开发可能理解成：登录后游客资产虽然不迁移，但也不再能访问。

你的需求应该是：登录已有账号不动游客资产；退出账号后还能看到原游客资产。

⸻

4. 「入库时间倒序」统一替换成「以第三方聚合数据为准」这句不太对

第十五部分冲突 3 写：

原 PRD 中出现的“入库时间倒序”统一替换为“以第三方聚合数据为准”。 

这里有点问题。

“入库时间倒序”是排序规则，而“以第三方聚合数据为准”是数据来源规则，两者不是一个维度，不能直接替换。

建议改成：

数据来源以当前 D1 卡牌基础表为准，App 不自行维护价格数据库。
Search / 系列列表等默认排序不再使用“客户端本地入库时间”作为业务口径，改为使用后端数据源适配层返回的默认排序字段，例如 updated_at、release_date 或采集程序维护的排序字段。具体排序字段由接口返回，前端按接口结果展示。

这样更准确。

6. 通用失败 Toast 文案有两个版本，要统一

你这个文件现在统一的是：

但前面部分页面里还出现过：

Something went wrong. Please try again later.

统一保留你 global-rules 里的版本：

Something went wrong. Please try again.

然后所有模块 PRD 遇到通用操作失败都引用 global，不再单独写 later 版本。

不过迁移失败、提交反馈失败、打开网页失败这类场景专用文案可以继续保留。

7. 「轻操作」定义里有一点拧巴

你现在写：

轻操作指不涉及高风险 / 不可逆后果的常规操作失败……也包含一般增删改保存。真正高风险操作不走通用 Toast。

但后面又把 删除文件夹失败 / 移除 Portfolio 失败 放进通用 Toast 适用按钮里。这个其实可以，但“轻操作”这个词容易引起歧义。

建议改成：

通用 Toast 适用于大多数可恢复的操作失败，包括切换、保存、添加、移除、删除等。操作失败时不改变数据状态，用户可重新操作。
账号删除、退出登录、官网链接打开、提交反馈、游客资产迁移等特殊场景使用专用文案。

不用强调“轻操作”，不然删除文件夹这种看起来不像轻操作。

8. 「删除账号失败」是否通用 Toast，要和 Profile 保持一致

你现在第十三部分专用失败文案里写：

Unable to complete this action. Please try again later.

用于删除账号失败 / 退出登录失败。这个没问题。

但第九部分确认弹窗规则又写：

确认失败后不改变数据，并展示通用 Toast。

建议改成：

确认失败后不改变数据，并按场景展示失败提示。普通删除 / 移除操作使用通用 Toast；删除账号、退出登录等账号级操作使用场景专用失败文案。

这样不会和专用文案表冲突。

9. 状态优先级里「局部模块加载失败」的位置可以接受，但要注意实现

现在状态优先级是：

1. 权限阻断
2. 整页加载失败
3. 页面加载中
4. 页面空状态
5. 页面正常数据状态
6. 局部模块加载失败
7. Toast

这个基本可以。

但局部失败不是整页状态，而是正常页面中的模块状态。为了避免误解，可以改成：

局部模块加载失败不是整页状态，不影响页面其他区域展示；它在页面正常数据状态内按模块独立展示。

不然开发可能以为局部失败优先级低于正常数据，所以不展示。

10. 游客资产迁移失败逻辑建议再严谨一点

你现在写：

迁移失败时，已注册账号仍保持登录态。 

这可以，但要补一条：迁移失败期间，不能让用户误以为资产已经迁移成功。

建议补：

迁移失败时，游客资产仍保留在 anonymous_account 下，不能标记为已迁移。新账号保持登录态，但页面需明确以账号资产为准展示；如需继续迁移游客资产，需要提供后台重试或前端重试入口。

如果你希望用户注册后立刻看到游客资产，那迁移失败时就不能直接切到空账号，否则用户会以为资产丢了。这里产品上建议：

注册成功但迁移失败时，优先保留当前游客资产展示，并提示稍后重试同步；迁移成功后再切换为正式账号资产。

这点可以让开发评估，但一定要避免“注册成功 → 迁移失败 → 用户看到空资产”。

建议你最终改的重点

最重要的出入就这 5 个：

1. 删除“未登录用户点击需要账号资产操作时调起登录 / 注册弹窗”，改为游客可正常产生资产。
2. 游客资产统一改为 anonymous_account 服务端备份，不再写“仅本地资产”。
3. 登录已有账号不迁移游客资产，但退出登录后应能恢复原 anonymous_account 游客资产。
4. “入库时间倒序”不要直接替换成“以第三方聚合数据为准”，排序规则和数据来源要分开写。
5. Search 列表明确只展示 30D Change 百分比，不展示涨跌金额。

其他内容基本和我们之前整理的口径一致。
